🧠 YAML Intelligents

Des templates qui décident, calculent et adaptent — pas de simples conversions d'unités.

Bascule automatique heat / off / cool

La T° extérieure pilote le mode via deux sensors de seuil — pas de valeurs hardcodées. seuil_non_chauffage (18°C) et seuil_non_clim (27°C) sont des sensors HA modifiables sans toucher au template. Le fallback -999 détecte une sonde hors ligne et retourne unknown au lieu de forcer un mode erroné.

✓ RésultatToutes les automations clim lisent sensor.mode_ete_hiver — un seul point de vérité. Modifier le seuil de bascule = changer la valeur d'un sensor, le template suit automatiquement.
P1_01_MASTER · réel
# --- mode_ete_hiver ---
- name: "Mode Été/Hiver"
unique_id: mode_ete_hiver
state: >
{% set t = states('sensor.th_balcon_nord_temperature') | float(-999) %}
{% set nc = states('sensor.seuil_non_chauffage') | float(18) %}
{% set ncl = states('sensor.seuil_non_clim') | float(28) %}
{% if t == -999 %} unknown {# sonde hors ligne #}
{% elif t < nc %} heat
{% elif t < ncl %} off
{% else %} cool
{% endif %}

Chaîne thermique — 6 sensors en cascade

La cible confort n'est pas un template unique. C'est une chaîne de 6 sensors qui se lisent les uns les autres. Chaque étape a un rôle clair et un unique_id propre — auditable et débuggable individuellement. Les paliers utilisent des sensors delta (delta_1…delta_7) au lieu de valeurs hardcodées.

✓ Chaîne complètemode_ete_hiver → consigne_de_base → temperature_differentielle → temperature_cible → temperature_confort_jour → temperature_confort_nuit
P1_01_MASTER · étape 4/6
# --- temperature_cible ---
- name: "Temperature Cible"
unique_id: temperature_cible
unit_of_measurement: "°C"
state: >
{% set t = states('sensor.th_balcon_nord_temperature') | float(0) %}
{% set base = states('sensor.consigne_de_base') | float(18) %}
{% set d1 = states('sensor.delta_1') | float(1) %}
{% set d5 = states('sensor.delta_5') | float(5) %}
{% set d7 = states('sensor.delta_7') | float(7) %}
{% if t < 12 %} {{ (base + d1) | int }} {# froid → +delta #}
{% elif t > 40 %} {{ (t - d7) | int }} {# canicule #}
{% elif t > 32 %} {{ (t - d5) | int }} {# forte chaleur #}
{% else %} {{ base | int }}
{% endif %}

Labels sémantiques — groupes Hue, pas lampes individuelles

Le template lit des groupes Hue natifs (light.salon, light.table), pas les ampoules une à une. 4 états propres : Allumé / Salon / Table / Éteint. Pour le bureau, la logique est plus complexe : 3 sensors imbriqués (lumiere_bureau_etat, lumiere_ecran_etat, bureau_etat) avec 6+ états selon la combinaison PC / Hue Sync / lampes.

✓ Résultat"Salon" = salon allumé, table éteinte. "Table" = table seule. Pour le bureau : "Allumé & Sync." = lampes ON + Hue Sync Box active. Dashboard toujours cohérent avec la réalité physique.
etats_status.yaml · réel
# --- lumiere_salon_etat ---
- name: "Lumière Salon État"
unique_id: lumiere_salon_etat
state: >
{% if is_state('light.salon', 'on') and is_state('light.table', 'on') %} Allumé
{% elif is_state('light.salon', 'off') and is_state('light.table', 'off') %} Éteint
{% elif is_state('light.salon', 'on') and is_state('light.table', 'off') %} Salon
{% elif is_state('light.salon', 'off') and is_state('light.table', 'on') %} Table
{% else %} erreur {% endif %}
# Bureau : 3 sensors imbriqués, 6 états possibles :
# Éco. / Écran / Allumé / Allumé & Sync. / Synchro. / Éteint
# Dépend de switch.ecran_p_c_3_play_hue + binary_sensor.moniteur_pc

Chaîne P4 → Absence → Switch → Card disparaît

La carte Écran (Hue Play) disparaît du dashboard quand switch.ecran_p_c_3_play_hue = off. Ce switch est coupé par une automation d'absence déclenchée depuis binary_sensor.presence_maison (P4). La carte Têtes de Lit + son heading entier disparaissent de la même façon via switch.prise_tete_de_lit_chambre.

P3 et P4 restent dans des fichiers YAML strictement séparés — c'est uniquement la conditional card du dashboard qui croise les pôles.

✓ Chaîne complèteP4 présence_maison = Absent → automation → switch OFF → conditional card → carte masquée. En parallèle : sensor.lumiere_ecran_etat passe à Éco. dans le badge Bureau.

🚫 Zéro Helper UI

Aucun input_boolean, input_number ou input_select pour la logique métier. Tout est en YAML — versionnable, auditable, rollbackable sur Git.

❌ Approche Helper UI
✗input_number.seuil_cool: 26.5 — modifiable librement dans l'UI HA
✗Valeur stockée en base HA — non versionnée sur Git
✗Diff illisible : "entity changed" sans contexte ni justification
✗Source de vérité : l'interface graphique — aucun audit possible
✗Impossible de savoir pourquoi le seuil a changé il y a 3 mois
✗Si HA crash et restore : les helpers reviennent à leur valeur par défaut
✅ Approche Template YAML
✓Seuils hardcodés dans la value_template: — dans le fichier YAML
✓Chaque modification = commit Git avec date et contexte
✓Diff lisible : t >= 26.5 → t >= 27 — intention claire
✓Source de vérité : TREE_CORRIGE/ — auditable sans HA
✓Rollback instantané si un seuil cause un problème
✓Restore HA : les seuils reviennent exacts — définis dans les fichiers
